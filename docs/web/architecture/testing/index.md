# 测试

> 测试是保障代码质量、降低回归风险的关键工程实践。本模块系统化介绍前端测试体系，涵盖 **单元测试 → 集成测试 → E2E 测试 → Mock/Stub → 测试覆盖率** 五大主题。

- **单元测试（Unit Test）**：验证单个函数 / 组件的行为，速度快、数量多。
- **集成测试（Integration Test）**：验证多个模块协作时的行为。
- **端到端测试（E2E Test）**：模拟真实用户操作，验证完整业务流程。

## 测试框架对比

| 框架        | 类型     | 特点                                   | 适用场景                |
| ----------- | -------- | -------------------------------------- | ----------------------- |
| **Jest**    | 单元/集成 | 开箱即用、Snapshot、Mock 功能强大      | React 项目、通用 JS 项目 |
| **Vitest**  | 单元/集成 | Vite 原生、ESM 支持、与 Jest API 兼容  | Vite 项目、现代 ESM 项目 |
| **Mocha**   | 单元/集成 | 灵活、可组合、需搭配断言库             | 定制化需求高的项目      |
| **Jasmine** | 单元/集成 | 自带断言与 Mock、行为驱动（BDD）       | Angular 项目            |
| **Cypress** | E2E      | 浏览器内运行、实时调试、可视化         | Web 应用 E2E 测试       |
| **Playwright** | E2E   | 跨浏览器、自动等待、多 Tab/多域支持    | 现代 Web 应用 E2E 测试  |
| **Testing Library** | 组件测试 | 以用户视角查询 DOM、不依赖实现细节 | React/Vue 组件测试      |

## 测试策略

### 1. 测试什么

- **核心业务逻辑**：工具函数、状态管理、数据转换。
- **组件交互**：用户输入、表单提交、状态切换。
- **边界条件**：空值、极值、异常输入。
- **关键流程**：登录、支付、下单等核心链路。

### 2. 不测试什么

- 第三方库的内部实现。
- 纯样式（除非影响交互）。
- 静态常量与配置。

### 3. 测试原则

- ** Arrange-Act-Assert（AAA）**：准备 → 执行 → 断言，结构清晰。
- **单一职责**：一个测试只验证一个行为。
- **独立可重复**：测试之间互不依赖，任意顺序执行结果一致。
- **快速反馈**：单元测试应在秒级完成。

## 推荐学习路线

```
1. 单元测试（Jest / Vitest）
   └─ 学习断言、Mock、Snapshot
2. 组件测试（Testing Library）
   └─ 学习以用户视角测试组件
3. 集成测试
   └─ 学习多模块协作测试
4. E2E 测试（Playwright / Cypress）
   └─ 学习模拟真实用户操作
5. 测试覆盖率
   └─ 学习度量与提升覆盖率
```
